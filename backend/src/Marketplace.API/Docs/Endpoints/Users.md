# UsersController — `/users`

Усі маршрути: **`[Authorize]`** (будь-який автентифікований користувач), окрім явно зазначених.

## `GET /users/me`

- **Summary (1 рядок):** Профіль поточного користувача з членствами в компаніях.
- **Призначення:** профіль маркетплейсу поточного користувача та список компаній з активним членством (`companyMemberships`: id, назва, slug, роль у компанії, `isOwner`).
- **Хто може викликати:**
  - JWT: обов'язково
  - Глобальні ролі: будь-який автентифікований користувач
  - Компанійні ролі: —
- **Бізнес-логіка:**
  1. Визначити `userId` з JWT claims
  2. Завантажити профіль та активні членства в компаніях
  3. Повернути `UserDto` з `companyMemberships`
- **Side effects (синхронно):** —
- **Async / «магія»:** —
- **Де на фронті:**
  - Екран: App shell / workspace context
  - API-модуль: `frontend/src/features/auth/api/auth.api.ts` (`getCurrentUser`)
  - Статус: `implemented`
- **Повертає:** **200** `UserDto`.
- **Помилки:** **401** якщо немає user id у claims.
- **Примітки:** компанії з soft-delete не потрапляють у `companyMemberships`. У відповідях `GET /users` та `GET /users/search` поле `companyMemberships` завжди порожній масив.

## `PATCH /users/me`

- **Summary (1 рядок):** Оновити персональні дані поточного користувача.
- **Призначення:** оновити `userName` та `phoneNumber` для поточного користувача.
- **Хто може викликати:**
  - JWT: обов'язково
  - Глобальні ролі: будь-який автентифікований користувач
  - Компанійні ролі: —
- **Бізнес-логіка:**
  1. Визначити `userId` з JWT claims
  2. Провалідувати `userName` і `phoneNumber` (правила як у реєстрації)
  3. Оновити профіль у domain + identity
  4. Інвалідувати кеш профілю користувача
- **Side effects (синхронно):** оновлення `User` в домені та `ApplicationUser` в Identity.
- **Async / «магія»:** —
- **Де на фронті:**
  - Екран: Account settings / profile
  - API-модуль: planned
  - Статус: `implemented`
- **Приймає:** body `{ "userName": "new_user_name", "phoneNumber": "+380501234567" }`.
- **Повертає:** **200** порожнє тіло при успіху.
- **Помилки:** **400** при невалідних полях, **401** без user id у claims, **404/Failure** якщо користувача не знайдено.

## `GET /users`

- **Summary (1 рядок):** Список усіх користувачів платформи.
- **Призначення:** список усіх користувачів з репозиторію.
- **Хто може викликати:**
  - JWT: обов'язково
  - Глобальні ролі: будь-який автентифікований (у коді **немає** обмеження роллю)
  - Компанійні ролі: —
- **Бізнес-логіка:**
  1. Завантажити всіх користувачів з репозиторію
  2. Повернути масив `UserDto` (без `companyMemberships`)
- **Side effects (синхронно):** —
- **Async / «магія»:** —
- **Де на фронті:**
  - Екран: Admin users (planned)
  - API-модуль: —
  - Статус: `planned`
- **Повертає:** **200** масив `UserDto`.
- **Примітки:** з точки зору продукту варто вважати це **чутливим** endpoint-ом; для адміністрування краще додати `Admin` policy на рівні контролера/політики.

- **Side effects (синхронно):** —
- **Async / «магія»:** —

## `DELETE /users/{id}`

- **Summary (1 рядок):** М'яке видалення власного акаунта.
- **Призначення:** м'яке видалення **власного** акаунта.
- **Хто може викликати:**
  - JWT: обов'язково
  - Глобальні ролі: власник акаунта (`id` = поточний user)
  - Компанійні ролі: —
- **Бізнес-логіка:**
  1. Перевірити, що `id` збігається з поточним користувачем
  2. Виконати soft-delete через `IUserManagementService`
- **Side effects (синхронно):** soft-delete профілю/акаунта
- **Async / «магія»:** —
- **Де на фронті:**
  - Екран: Account settings / delete account (planned)
  - API-модуль: —
  - Статус: `planned`
- **Приймає:** path `id` (guid) — має збігатися з поточним користувачем.
- **Повертає:** **200** порожнє тіло.
- **Помилки:** **403** якщо `id` ≠ поточний user; **401** без user id.

## `PATCH /users/{id}/role`

- **Summary (1 рядок):** Призначення глобальної ролі користувачу (Admin only).
- **Призначення:** призначити **глобальну** роль маркетплейсу (`UserRole`: `Buyer`, `Seller`, `Moderator`, `Admin` — у body як рядок, case-insensitive).
- **Хто може викликати:**
  - JWT: обов'язково
  - Глобальні ролі: **Admin**
  - Компанійні ролі: —
- **Бізнес-логіка:**
  1. Перевірити роль Admin у JWT
  2. Валідувати `role` у body
  3. Оновити глобальну роль через `IAuthenticationPort`
- **Side effects (синхронно):** оновлення ролі в Identity/порті автентифікації
- **Async / «магія»:** —
- **Де на фронті:**
  - Екран: AdminShell / user management (planned)
  - API-модуль: —
  - Статус: `partial`
- **Приймає:** path `id`; body `{ "role": "buyer|seller|moderator|admin" }` (`AssignUserRoleRequest`).
- **Повертає:** **200** порожнє тіло при успіху (`Result` без значення).
- **Помилки:** **400** невалідна `role`; інші — з `IAuthenticationPort`.
